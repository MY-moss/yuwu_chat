## 二、AI 调用标准流程

### 2.1 接收任务时（必做）

按以下顺序读取文档：

```
1. AI工作流指南.md          ← 了解文档体系和流程（本文件）
2. 项目架构和相关规范.md     ← 了解架构、目录、修改规则、待办优先级
3. 问题反馈与报告.md         ← 了解现存问题（若涉及修复）
4. 代码分类索引.md           ← 定位代码区块（若涉及代码修改）
5. 优化建议.md               ← 了解待审批优化（若用户要求优化）
```

**基线评估**（可选但推荐）：
```bash
python docs/AI行为评估脚本.py --auto
```

### 2.2 执行任务时（必遵守）

| 规则 | 说明 |
|------|------|
| **唯一修改入口** | 所有代码修改必须在 `src/backend/` 目录进行 |
| **同步规则** | 修改完成后同步到 `release/`（前端文件及数据文件） |
| **禁止操作** | 直接在 `release/` 中修改代码 |
| **分区注释** | 新增功能时确定所属区块，更新文件头区块数 |
| **优化建议** | 需用户明确审批后才能执行，AI 无权自行执行 |
| **端口固定** | 必须为 9000 |
| **JSON 安全** | JSON 文件操作必须使用 threading.Lock() 保护 |
| **CSRF** | 所有 POST/PUT/DELETE 请求必须包含 CSRF token（登录/注册除外） |

### 2.3 完成任务时（必检查）

**代码验证**：
```bash
# 1. Python 语法检查
python -c "import ast; ast.parse(open('src/backend/app.py', encoding='utf-8').read())"

# 2. 区块数验证（app.py=22, script.js=18, style.css=9, feedback.js=9, feedback.css=6）
grep -c "区块" src/backend/app.py
grep -c "// 区块" src/backend/static/script.js

# 3. END 标记验证
grep "END OF FILE" src/backend/app.py src/backend/static/*.js src/backend/static/*.css
```

**文档更新检查清单**（按顺序）：
- [ ] `问题反馈与报告.md` — 若修复了问题，从现存清单移除
- [ ] `已修复历史问题.md` — 若修复了问题，添加修复记录
- [ ] `项目架构和相关规范.md` — 若版本/架构/规则变更，更新对应章节
- [ ] `代码分类索引.md` — 若新增/删除区块，更新区块表
- [ ] `优化建议.md` — 若执行了优化项，更新状态为已执行
- [ ] `AI工作流指南.md` — 若文档体系变更，更新清单

**终态评估**：
```bash

```

## 三、文档同步规则

### 3.1 权威来源机制

每类信息**仅在一个文件中维护**，其他文件引用而非重复：

| 信息类型 | 权威文件 | 其他文件做法 |
|---------|---------|------------|
| 项目定位/技术栈/端口/版本 | 项目架构和相关规范.md | 引用，不重复 |
| 目录结构/关键路径 | 项目架构和相关规范.md | 引用 |
| 修改规则/AI接手规则 | 项目架构和相关规范.md + 本文件 | 引用 |
| 版本历史/变更记录 | 项目架构和相关规范.md | 引用 |
| 现存问题清单 | 问题反馈与报告.md | 引用数字，不展开 |
| 已修复问题档案 | 已修复历史问题.md | 引用累计数 |
| 代码区块结构 | 代码分类索引.md | 引用区块数 |
| AI 行为评估标准 | AI行为评估脚本.py | 引用 |
| 待办优先级/已知风险 | 项目架构和相关规范.md（七、） | — |
| 待审批优化建议 | 优化建议.md | — |
| AI 工作流/文档使用 | 本文件 | — |

### 3.2 同步顺序

每次审计或修改后，按以下顺序更新文档：
1. `问题反馈与报告.md` — 先更新问题清单
2. `已修复历史问题.md` — 添加修复记录
3. `项目架构和相关规范.md` — 更新版本历史、同步状态、待办/风险
4. `代码分类索引.md` — 更新区块表（若涉及代码结构变更）
5. `优化建议.md` — 更新优化项状态（若执行了优化）

### 3.3 版本号规则

| 变更类型 | 版本号变化 | 示例 |
|---------|-----------|------|
| 代码结构优化/架构变更 | minor +1 | v2.1.4 → v2.1.5 |
| 问题修复 | patch +1 | v2.1.5 → v2.1.6 |
| 重大功能迭代 | major +1 | v2.1.5 → v3.0.0 |
